# DATA ARCHITECTURE SPEC

Canonical data architecture for the Communications Intelligence Platform.

Status: Final data-model blueprint before rebuilding integrations.

Last updated: 2026-06-23

## 1. Architecture Principles

1. Airtable remains the operational source of truth for dashboard-ready analytics.
2. `KPI_History` is the canonical KPI calculation ledger.
3. `Channel_Performance` is the canonical channel-level dashboard ledger.
4. External integrations normalize into a shared Data Source Metric model before writing dashboard-ready rows.
5. Dashboard pages must not depend on raw API payloads, CSV import artifacts, or connector-specific field names.
6. Formula, lookup, rollup, created-time, and last-modified fields may be present for Airtable convenience, but integrations must write only editable fields.
7. Every writable analytics table must have a stable `Unique Key` so syncs can upsert without duplicates.

## 2. Time and Aggregation Rules

### Supported Periods

Use `Period Type` with one of:

- `Daily`
- `Weekly`
- `Monthly`
- `Quarterly`
- `Yearly`
- `Custom`

### Canonical Date Semantics

- `Date` is required and must equal `Period Start`.
- `Period Start` is required.
- `Period End` is required.
- Weekly periods use Monday through Sunday.
- Monthly periods use calendar months.
- Quarterly periods use calendar quarters.
- Custom ranges use the exact selected start and end dates.

### Dashboard Time Window Logic

Weekly:

- Read rows where `Period Type = Weekly`.
- Include rows where `Period Start` and `Period End` fall inside the selected week.
- If weekly rows do not exist, aggregate daily rows inside the week.

Monthly:

- Read rows where `Period Type = Monthly`.
- Include rows where `Period Start` and `Period End` fall inside the selected calendar month.
- If monthly rows do not exist, aggregate weekly or daily rows inside the month.

Custom Range:

- Prefer rows where `Period Type = Custom` and the row exactly matches the selected range.
- Otherwise aggregate daily rows inside the range.
- If only weekly or monthly rows exist, include rows whose `Period Start` is inside the selected range. Do not prorate partial periods unless a future integration explicitly supports prorated values.

### Aggregation Methods

Each metric row must declare `Aggregation Method`:

- `Sum`: counts and volume metrics, such as clicks, streams, posts, visitors.
- `Average`: simple average where each row is equally weighted.
- `Weighted Average`: rates where `Numerator` and `Denominator` are available.
- `Latest`: snapshot values, such as current subscribers.
- `Calculated`: derived metrics generated by the KPI engine.

Rates should use weighted averages whenever possible:

```text
Weighted Rate = sum(Numerator) / sum(Denominator) * 100
```

If numerator and denominator are unavailable, use average of submitted rate values and mark `Quality Status = Estimated`.

## 3. Canonical Tables

### 3.1 KPI_History

Purpose:

`KPI_History` is the source of truth for KPI Overview, KPI score calculations, growth indicators, trend charts, and Monthly Snapshot.

Grain:

One row per metric, channel, platform, source, and period.

Required editable fields:

| Field | Type | Required | Description |
| --- | --- | --- | --- |
| `Unique Key` | Single-line text | Yes | Stable upsert key. |
| `Metric` | Single select or text | Yes | Canonical metric name used by the KPI engine. |
| `Metric Key` | Single-line text | Yes | Stable machine key, e.g. `email_click_rate`. |
| `Value` | Number | Yes | Numeric value for the metric. Percent values are stored as `0-100`, not `0-1`. |
| `Unit` | Single select or text | Yes | `count`, `percent`, `score`, `items`, `streams`, `clicks`, etc. |
| `Period Type` | Single select | Yes | `Daily`, `Weekly`, `Monthly`, `Quarterly`, `Yearly`, or `Custom`. |
| `Date` | Date | Yes | Same as `Period Start`; retained for current dashboard compatibility. |
| `Period Start` | Date | Yes | Inclusive start date. |
| `Period End` | Date | Yes | Inclusive end date. |
| `Aggregation Method` | Single select | Yes | `Sum`, `Average`, `Weighted Average`, `Latest`, or `Calculated`. |
| `Channel` | Single select | Yes | Canonical channel group: `Email`, `Social`, `Podcast`, `Website`, or `Corporate`. |
| `Platform` | Single select or text | Yes | Source platform, e.g. `Mailchimp`, `Instagram`, `Spotify`. |
| `Source Name` | Single select or text | Yes | Integration/source that produced the row. |
| `Quality Status` | Single select | Yes | `Complete`, `Estimated`, `Partial`, `Missing`, or `Error`. |

Recommended editable fields:

| Field | Type | Description |
| --- | --- | --- |
| `KPI` | Text | Human-friendly display label. Can mirror `Metric`. |
| `Numerator` | Number | Numerator for weighted rates. |
| `Denominator` | Number | Denominator for weighted rates. |

| `Source Reliability Score` | Number | Optional `0-100` confidence score. |
| `Notes` | Long text | Data quality notes. |

## 4. KPI Engine Input Mapping

The KPI engine reads canonical metric rows from `KPI_History`.

General row lookup:

```text
Metric = required metric name
Value = current value
Unit = metric unit
Date / Period Start / Period End = time window
```

Previous value logic:

1. Prefer an explicit `Previous Value` on the current row if present.
2. Otherwise find the same `Metric Key`, `Channel`, and `Platform` in the previous equivalent period.
3. For weekly comparison, previous equivalent period is the prior week.
4. For monthly comparison, previous equivalent period is the prior calendar month.
5. For custom range, previous equivalent period is the immediately preceding range of the same length.

### Communications Performance Score

Formula:

```text
(Email Performance Score + Social Media Performance Score + Podcast Performance Score + Website Performance Score) / 4
```

Reads:

- Derived from the four channel scores below.
- Optional persisted snapshot row: `Metric = Communications Performance Score`, `Metric Key = communications_performance_score`.

Aggregation:

- Calculated for the selected time window from component metric rows.
- If persisted, use `Aggregation Method = Calculated`.

### Email Performance Score

Formula:

```text
(Click Rate score * 40%)
+ (Open Rate score * 30%)
+ (Subscriber Growth score * 20%)
+ (Unsubscribe Rate Health score * 10%)
```

Reads from `KPI_History`:

| Engine Input | Metric | Metric Key | Unit | Aggregation |
| --- | --- | --- | --- | --- |
| Click Rate | `Email Click Rate` | `email_click_rate` | `percent` | Weighted Average |
| Open Rate | `Email Open Rate` | `email_open_rate` | `percent` | Weighted Average |
| Subscriber Growth | `Subscriber Growth` | `subscriber_growth` | `percent` | Calculated |
| Unsubscribe Rate | `Unsubscribe Rate` | `unsubscribe_rate` | `percent` | Weighted Average |

Aggregation details:

- `Email Click Rate`: use `sum(clicks) / sum(emails_delivered) * 100` when numerator/denominator are available.
- `Email Open Rate`: use `sum(opens) / sum(emails_delivered) * 100`.
- `Subscriber Growth`: use `(current subscribers - previous subscribers) / abs(previous subscribers) * 100`, or use submitted percent if integration provides it.
- `Unsubscribe Rate`: use `sum(unsubscribes) / sum(emails_delivered) * 100`; lower is better.

### Social Media Performance Score

Formula:

```text
(Posts Published score * 20%)
+ (Likes score * 30%)
+ (Comments score * 25%)
+ (Shares score * 25%)
```

Reads from `KPI_History`:

| Engine Input | Metric | Metric Key | Unit | Aggregation |
| --- | --- | --- | --- | --- |
| Posts Published | `Posts Published` | `posts_published` | `items` | Sum |
| Likes | `Likes` | `likes` | `count` | Sum |
| Comments | `Comments` | `comments` | `count` | Sum |
| Shares | `Shares` | `shares` | `count` | Sum |

Aggregation details:

- Aggregate across rows where `Channel = Social`.
- Supported current social platforms are `Instagram` and `Facebook`.
- Future platforms can join by writing `Channel = Social`.

### Podcast Performance Score

Formula:

```text
(Listener Growth score * 40%)
+ (Streams Per Episode score * 40%)
+ (Streams score * 20%)

Streams Per Episode = Streams / Podcasts Published
```

Reads from `KPI_History`:

| Engine Input | Metric | Metric Key | Unit | Aggregation |
| --- | --- | --- | --- | --- |
| Streams | `Streams` | `streams` | `streams` | Sum |
| Podcasts Published | `Podcasts Published` | `podcasts_published` | `items` | Sum |
| Listener Growth | `Listener Growth` | `listener_growth` | `percent` | Calculated |

Aggregation details:

- Aggregate across rows where `Channel = Podcast`.
- `Spotify` writes streams.
- `Castos` writes downloads to `Channel_Performance`; only map Castos into KPI `Streams` if the organization decides downloads count as podcast consumption for the score.

### Website Performance Score

Formula:

```text
(Visitors score * 40%)
+ (Clicks score * 35%)
+ (Engagement score * 25%)
```

Reads from `KPI_History`:

| Engine Input | Metric | Metric Key | Unit | Aggregation |
| --- | --- | --- | --- | --- |
| Visitors | `Website Visitors` | `website_visitors` | `count` | Sum |
| Clicks | `Website Clicks` | `website_clicks` | `clicks` | Sum |
| Engagement | `Website Engagement` | `website_engagement` | `percent` | Weighted Average |

Aggregation details:

- Aggregate across rows where `Channel = Website`.
- `Website Engagement` should use weighted average when numerator and denominator exist.

## 5. Growth Indicator Mapping

Growth indicators read from `KPI_History` using current and previous equivalent periods.

| Dashboard Indicator | Metric | Metric Key | Unit | Aggregation |
| --- | --- | --- | --- | --- |
| New Followers | `New Followers` | `new_followers` | `count` | Sum |
| New Subscribers | `New Subscribers` | `new_subscribers` | `count` | Sum |
| New Podcast Listeners | `New Podcast Listeners` | `new_podcast_listeners` | `count` | Sum |
| Website Visitors | `Website Visitors` | `website_visitors` | `count` | Sum |
| Content Published | `Content Published` | `content_published` | `items` | Sum |

Growth formula:

```text
Growth % = (Current Period Value - Previous Period Value) / abs(Previous Period Value) * 100
```

If previous value is missing or zero:

- If current value is positive and previous is zero, display `100%`.
- If previous value is missing, display no comparison.

## 6. Dashboard Compatibility

### KPI Overview

Works from:

- `KPI_History`
- `Monthly_Activity_Summary`
- `Data_Source_Status`
- `Spotify_Weekly_Snapshot` only as a backward-compatible trend source

Required behavior:

- Executive Summary is derived from `KPI_History` metric coverage and source status.
- Performance Scores are calculated from `KPI_History`.
- Growth Indicators are calculated from `KPI_History`.
- Monthly Snapshot is grouped from `KPI_History` by week within the selected month/range.
- Monthly Activity Summary should read `Monthly_Activity_Summary`; if absent, it can be derived from `KPI_History` output metrics.
- Charts read time-series values from `KPI_History`; Spotify snapshot remains a secondary fallback only.

### Channel Breakdown

Works from:

- `Channel_Performance`

Required behavior:

- One card per supported channel.
- `Metric Value` is the primary number shown.
- `Activity Volume` is the secondary output volume.
- `Change Percent` can be read directly or calculated against previous equivalent period rows.
- Trend chart is generated from historical `Channel_Performance` rows.

### Top Performing Content

Works from:

- `Content_Performance`

Required behavior:

- Rankings are read from Airtable fields, not calculated in backend.
- Sort by `Rank` first.
- If `Rank` is missing, sort by `Content Score`.
- If both are missing, display the content row but show `No data available` for score.
- Grouping can use `Campaign`, `Episode Drop`, or `Marketing Push`.

## 8. Implementation Notes for Airtable

Recommended table order:

1. `KPI_History`
2. `Channel_Performance`
3. `Content_Performance`
4. `Monthly_Activity_Summary`
5. `Data_Source_Status`
6. Optional staging table: `Data_Source_Metrics`

Recommended Airtable field constraints:

- Use single select fields for `Period Type`, `Channel`, `Metric Type`, `Aggregation Method`, and `Quality Status`.
- Use number fields for all metric values.
- Store rates as `0-100`.
- Keep `Unique Key` as editable text, not a formula field, if integrations will upsert records.
- Keep display/helper formula fields separate from writable fields.

## 9. Definition of Done Before Rebuilding Integrations

Before adding live integrations:

1. Create or update Airtable fields to match this spec.
2. Confirm `Unique Key` is editable in each writable table.
3. Confirm all required select options exist.
4. Add one manual weekly row and one manual monthly row for each core KPI metric.
5. Add one manual row per supported channel in `Channel_Performance`.
6. Add at least one `Content_Performance` row with `Rank` and `Content Score`.
7. Run `/api/data-health`.
8. Resolve missing required fields or empty critical tables.
9. Only then rebuild Mailchimp, Spotify, Castos, YouTube, Website, Facebook, and Instagram integrations.
